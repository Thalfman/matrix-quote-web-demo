# Plan 04-03 — Shared jargon-guard module + test (PLAN VIEW)

> Two-view plan. **PLAN VIEW** for human review. **CODE VIEW** below for the executor. Adds a new test surface; does NOT modify any production component.

## Why this plan

DATA-03 says the jargon-guard test must extend to scan `QuoteResultPanel.tsx`, `BusinessInsights*.tsx`, and any new copy in those surfaces. Today the jargon-guards live inline in eight different test files (the canonical regex set sits at `glossary.test.ts:63-80`); the three named surfaces have NO jargon-guard at all (BusinessInsightsView has one but only on the *download surface*, not the page body).

The Phase 3 R5 risk note in `03-05-PLAN.md` explicitly forecast this lift: *"Phase 4 will lift them into a single shared `jargon-guard.test.ts`."* Phase 4 honors that forecast in two parts:

1. New `frontend/src/test/jargon.ts` — exports `BANNED_TOKENS` as the single source of truth (copy of the array currently inlined in `glossary.test.ts`).
2. New `frontend/src/test/jargon-guard.test.tsx` — renders each previously-uncovered surface (`<QuoteResultPanel>`, `<BusinessInsights>`, `<BusinessInsightsView>`) and runs the regex set over `document.body.textContent`.

Plus one mechanical refactor: `glossary.test.ts` swaps its inline `BANNED` const for `import { BANNED_TOKENS }`. Pure dedup; the assertions stay identical.

The other inline jargon-guards (in `DataProvenanceNote.test.tsx`, `DemoHome.test.tsx`, `Tooltip.test.tsx`, `MachineLearningQuoteTool.test.tsx`, `buildBundleReadme.test.ts`, `buildPortfolioWorkbook.test.ts`, `BusinessInsightsView.test.tsx`) are NOT touched. Their inline regex sets are subsets of the canonical list, often paired with file-specific assertions; refactoring them risks accidentally narrowing coverage. CONTEXT D-03 locks this trade-off; future opportunistic dedup is welcome but out of scope here.
