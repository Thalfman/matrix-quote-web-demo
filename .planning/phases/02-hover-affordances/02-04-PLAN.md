# Plan 02-04 — Recap Headings + Form Label Glossary (PLAN VIEW)

> Two-view plan. PLAN VIEW for human review; CODE VIEW for the executor.

## Why this plan

This plan completes UX-03 by wrapping the customer-facing labels Ben sees on the Quote screens — recap section headings (`QuoteResultPanel.tsx`) and form field labels (`QuoteForm.tsx`) — with the project glossary tooltip from plan 02-01. Like 02-03 it depends on Wave 1 (`Tooltip.tsx` + `glossary.ts`).

It runs PARALLEL to 02-03 in Wave 2 because the file sets do not overlap (chart files vs quote-panel/form files).

The non-trivial part is the `Field` component. Today, `Field` accepts `label: string`. We add an optional `glossaryTerm?: string` prop. When provided, the label text is wrapped with `<Tooltip term={glossaryTerm}>` + `<GlossaryHelpIcon>`. When omitted, the rendered label is byte-identical to today (no spurious affordance). This contains the change to one call site (Field), then we sprinkle the prop on the eight matching Fields in `QuoteForm`.

For `QuoteResultPanel`, we wrap each `SECTIONS` heading conditionally: if `lookup(section.title)` returns an entry, render the heading inside a `<Tooltip>` with a focusable proxy; otherwise render the plain heading. This honors CONTEXT D-07 ("Section titles without matching terms stay plain"). NB: of the six section titles, NONE of them match the 8 glossary keys verbatim — the section titles are "Project classification", "Physical scale", "Controls & automation", etc., not "System Category" / "Sales Bucket" / etc. The MATCHING surface is the per-row LABELS within each section (e.g., "Industry segment", "System category", "Vision type"). We extend the lookup to these row labels too, with a normalized-key map so we don't have to add lowercase variants to glossary.ts.

## Risks / pitfalls

- **R1 — `Field` consumers throughout the codebase.** Search reveals `Field` is imported in QuoteForm + a handful of admin/full-app pages (vestigial). Adding an OPTIONAL prop is non-breaking. Confirm no consumer passes a `glossaryTerm` prop before this PR (none should — the prop is brand new).
- **R2 — Casing mismatch between Field labels and glossary keys.** QuoteForm uses sentence-case ("Industry segment") and the glossary keys use Title Case ("Industry Segment"). Use a per-call-site `glossaryTerm="Industry Segment"` prop — do NOT auto-derive from the label string. Explicit is better here so the executor knows exactly which 8 fields are in scope.
- **R3 — `Field` label is currently a `string`, not a ReactNode.** Today: `<label className="block text-xs eyebrow text-muted mb-1.5">{label}</label>`. We keep the `label: string` prop type and conditionally render the inner content as `<Tooltip>{label}</Tooltip>` when `glossaryTerm` is set — no signature break.
- **R4 — Layout shift.** Adding a `<HelpCircle>` icon next to a label might push form widths. Use `inline-flex items-center gap-1` so the icon sits inside the label baseline. The icon is 12px, gap 4px — negligible.
- **R5 — `<TooltipProvider>` mounting.** Each `<Tooltip>` in `Field` is wrapped in its own `<TooltipProvider>` so Field is self-contained. Same approach as plan 02-03's chart cards. Multiple providers nest fine.
- **R6 — `QuoteResultPanel.tsx` SECTIONS row labels.** The `SECTIONS` const (lines 173-242) lists row labels like `"Industry segment"`, `"System category"`, `"Automation level"`, `"PLC family"`, `"HMI family"`, `"Vision type"`, `"Overall complexity (1–5)"`. We map these to glossary keys via a small lookup helper that normalizes to Title Case for the 8 known terms. "Overall complexity (1–5)" maps to "Complexity (1–5)" — the canonical glossary key.
- **R7 — Test churn in QuoteResultPanel.test.tsx.** The Phase 1 test asserts on text presence using `screen.getByText(...)`. Adding a `<Tooltip>` wrapper around the heading does NOT change the rendered text (Radix `<TooltipTrigger asChild>` passes through the child). However, our `<Tooltip>` wrapper renders a `<button>`-style proxy (HelpCircle) NEXT to the label, so the text-by-itself node still exists and the existing assertions still pass. Spot-check after T3.
- **R8 — Existing UX-01 tests in QuoteResultPanel.test.tsx.** Phase 1 added tests for "Industry segment" / "System category" / etc. row labels. Those existing assertions still pass after T3 — Radix preserves the text node. Verify with `npm test` after the change.
- **R9 — Don't over-tooltip.** Only the eight Field labels listed in CONTEXT D-07 get a tooltip. NOT every Field. Adding tooltips to numeric counts ("Stations count", "Robot count", etc.) is OUT OF SCOPE per CONTEXT D-08.

## Out of scope

- [ ] No edits to chart files (plan 02-03).
- [ ] No tooltip on KPI cards, table headers, breadcrumbs, advanced-section indices (D-08 deferred).

**Diff plan:**

2. Extend the props type:

   ```ts
   type FieldProps = {
     label: string;
     hint?: string;
     error?: string;
     /** Optional glossary term key. When set, the label gets a focusable HelpCircle that opens the glossary tooltip. */
     glossaryTerm?: string;
   ```

3. Update the rendered label:

   ```tsx
   export function Field({ label, hint, error, glossaryTerm, children }: FieldProps) {
     const labelContent = glossaryTerm ? (
       <span className="inline-flex items-center gap-1">
         <span>{label}</span>
         <TooltipProvider delayDuration={200}>
           <Tooltip term={glossaryTerm} side="top">
             <GlossaryHelpIcon ariaLabel={`What is ${glossaryTerm}?`} />
           </Tooltip>
         </TooltipProvider>
       </span>
     ) : (
       label
     );

     return (
       <div>
         <label className="block text-xs eyebrow text-muted mb-1.5">{labelContent}</label>
         {children}
         {hint && <p className="text-xs text-muted mt-1">{hint}</p>}
         {error && <p className="text-xs text-danger mt-1">{error}</p>}
       </div>
     );
   }
   ```

   Key invariant: when `glossaryTerm` is undefined, the rendered HTML for the label is byte-identical to today.

   ```ts
   /**
    * Resolve a recap row label to a glossary key, if one exists. Returns
    * null when the row has no matching glossary entry — the row renders
    * plainly (no tooltip).
    */
   function recapLabelToGlossaryTerm(label: string): string | null {
     // Map of recap row labels (left-hand-side `<dt>`) to their glossary keys.
     // Only the 8 in CONTEXT D-04 are present; all other rows render plain.
     const MAP: Record<string, string> = {
       "Industry segment": "Industry Segment",
       "System category": "System Category",
       "Automation level": "Automation Level",
       "PLC family": "PLC Family",
       "HMI family": "HMI Family",
       "Vision type": "Vision Type",
       "Overall complexity (1–5)": "Complexity (1–5)",
     };
     const candidate = MAP[label];
     if (candidate && lookup(candidate)) return candidate;
     return null;
   }
   ```
