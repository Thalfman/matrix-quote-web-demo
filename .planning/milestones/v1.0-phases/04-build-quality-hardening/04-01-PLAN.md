# Plan 04-01 — vercel.json cache rule fix (PLAN VIEW)

> Two-view plan. **PLAN VIEW** for human review. **CODE VIEW** below for the executor. Pure config edit; no test changes.

## Why this plan

Today `vercel.json:11` caches `/demo-assets/models/(.*)` — a path that does not exist in the deployed site. The actual joblib bundles ship under `/demo-assets/models_real/` and `/demo-assets/models_synthetic/` (per `scripts/build_demo_static.py:_copy_model_bundle`). Result: every browser that loads the ML tool re-downloads the joblibs on every page load. The bundles are ~5-50 MB each. CDN cost + page-load regression. None of it visible to Ben yet because the demo is single-user, but it bites the moment usage scales.

DATA-01 fixes this by replacing the dead rule with two real-path rules. Both bundles are content-addressed by the build (filenames are recomputed from `core.config.TARGETS` on each `python3 scripts/build_demo_static.py` run), so the existing `immutable` directive on the dead rule is correct for the new rules too.

Pure config edit. No test changes. No deploy.

## Risks / pitfalls

- **R1 — Vercel rule precedence.** Vercel matches `headers[]` top-down on first hit. Since all three rules have disjoint `source` patterns, ordering doesn't change behavior. Keep the `models_real → models_synthetic → py` order anyway (CONTEXT D-01) for readability.
- **R2 — Trailing comma.** vercel.json is strict JSON (not JSON5). After deleting the dead object and inserting two new ones, double-check no orphan comma slipped in.
- **R3 — Don't introduce a new top-level key.** The phase only edits the `headers[]` array and nothing else. `buildCommand`, `outputDirectory`, `installCommand`, `framework`, `rewrites` stay byte-identical.
- **R4 — Vercel re-deploy not in scope.** This commit lands the file change; the actual cache-rule activation happens on the next deploy (`/gsd-ship 4` step, after Phase 4 verification).

# Plan 04-01 — CODE VIEW

## T1 — Edit vercel.json

- `vercel.json` (the file being modified — see current state)
- `.planning/REQUIREMENTS.md` § DATA-01 (acceptance criteria)
- `.planning/phases/04-build-quality-hardening/04-CONTEXT.md` § D-01 (locked decisions)

**Action:**

Replace the existing single object inside `headers[]` (the one with `"source": "/demo-assets/models/(.*)"`) with TWO objects, in this order: `models_real` first, then `models_synthetic`. The `/demo-assets/py/(.*)` object stays unchanged in third position.

Final `headers[]` MUST read exactly:

```json
"headers": [
  {
    "source": "/demo-assets/models_real/(.*)",
    "headers": [
      { "key": "Cache-Control", "value": "public, max-age=31536000, immutable" }
    ]
  },
  {
    "source": "/demo-assets/models_synthetic/(.*)",
    "headers": [
      { "key": "Cache-Control", "value": "public, max-age=31536000, immutable" }
    ]
  },
  {
    "source": "/demo-assets/py/(.*)",
    "headers": [
      { "key": "Cache-Control", "value": "public, max-age=3600" }
    ]
  }
]
```

Indentation: 2 spaces (matches the rest of the file). Do NOT reformat outside the `headers[]` array.

**Acceptance criteria (verifiable):**

- `node -e "const v=JSON.parse(require('fs').readFileSync('vercel.json','utf8')); const sources=v.headers.map(h=>h.source); if(JSON.stringify(sources)!==JSON.stringify(['/demo-assets/models_real/(.*)','/demo-assets/models_synthetic/(.*)','/demo-assets/py/(.*)'])) throw new Error('order mismatch: '+JSON.stringify(sources));"` exits 0.
