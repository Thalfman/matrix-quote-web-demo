## Risks / pitfalls

- **R1 — Template drift.** The README template is reproduced verbatim from CONTEXT D-06. Changing wording in this plan triggers Phase 4 DATA-03 to scan it for jargon — keep the wording identical to D-06. Snapshot test in T2 is the trip wire.
- **R2 — `generatedAt` ISO formatting.** Match `buildSummaryMarkdown`'s pattern (`generatedAt.toISOString()`) so the README's `Generated:` line is consistent with `summary.md`. Don't switch to a locale-aware formatter — ISO is unambiguous and timezone-stable.
- **R3 — `buildPortfolioJson` byte equivalence.** The PRIOR bundle wrote `JSON.stringify(portfolio, null, 2)` (see `exportPack.ts:133`). The new helper MUST produce the exact same bytes given the exact same input. Two-space indent, default key ordering (insertion order on `PortfolioStats`), no trailing newline. T4 asserts byte equivalence by comparing against the literal `JSON.stringify(portfolio, null, 2)` value.
- **R4 — No XLSX dependency in this plan.** The README is plain markdown; the JSON is plain `JSON.stringify`. Neither helper touches `xlsx`. This plan is therefore independently buildable even if `xlsx` install in 03-01 hits a snag — it only depends on Wave 1 completing as a whole before Wave 2 starts (the orchestrator coordinates).
- **R5 — Zero ML jargon in README.** The template names "Pearson r" indirectly via the workbook's Drivers tab — but the README does NOT say "Pearson". It says "Spreadsheet with four tabs: Summary, Drivers, Raw, README." Plain-English names. T2 jargon-guards the actual produced string.
- **R6 — Engineer-side button copy.** D-07 locks the wording: `Download raw JSON (for engineers)`. The README mentions this in the "Need the raw data?" block. Keep the parenthetical exactly as written — that's the disclaimer.
- **R7 — `summary.md` is locked (CONTEXT D-05).** This plan does NOT modify `buildSummaryMarkdown`. The new helpers append at the bottom of `exportPack.ts`.
- **R8 — Idempotent XLSX import (Wave 1 coordination with 03-01).** Plan 03-01 inserts `import * as XLSX from "xlsx";` at the TOP of `exportPack.ts`. This plan **does not need** the `xlsx` import — `buildBundleReadme`, `buildPortfolioJson`, and `jsonFilename` are pure markdown / pure `JSON.stringify` / pure string slugify. Do not add the import; if the worktree shows the import already present, leave it alone (do not duplicate, do not remove).
- **R9 — `jsonFilename` slug parity with `packFilename`.** The slug rule is identical (lowercase, non-alphanum → `-`, trim leading/trailing `-`, `"pack"` fallback for empty). Reused inline rather than extracted to a private helper because (a) it's six lines, (b) extraction would require a third edit point and (c) `packFilename` already exists; future refactor can DRY both helpers if a third caller appears. T4 covers the empty-label fallback so any drift trips a test.

```ts
// ---------------------------------------------------------------------------
// Bundle README template + engineer-side JSON helpers (CONTEXT D-06, D-07, D-08)
//
// buildBundleReadme produces the top-level README.md the zip ships with.
// Hand-written; substitutes only {datasetLabel} and {ISO date}.
// Names the three default-bundle files (summary.md, business-insights.xlsx,
// README.md) and points engineers to the secondary "Download raw JSON
// (for engineers)" button per CONTEXT D-07.
//
// buildPortfolioJson is the engineer-side download body. Byte-equivalent to
// the prior bundle's portfolio.json so engineers' downstream tooling keeps
// working without changes.

/**
 * Top-level README.md that ships inside the insights pack zip.
 *
 * Template per CONTEXT D-06; only {datasetLabel} and {ISO date} are
 * substituted at build time. Zero ML jargon — INSIGHTS-02.
 */
export function buildBundleReadme(
  datasetLabel: string,
  generatedAt: Date = new Date(),
): string {
  return [
    `# Business Insights Pack — ${datasetLabel}`,
    "",
    `Generated: ${generatedAt.toISOString()}`,
    "",
    "## What's inside",
    "- `summary.md` — One-page narrative summary of this dataset.",
    "- `business-insights.xlsx` — Spreadsheet with four tabs: Summary, Drivers, Raw, README.",
    "- `README.md` — This file.",
    "",
    "## Where to start",
    "- Open `summary.md` in any text editor or your browser. (Try VS Code, Notepad, or just double-click.)",
    "- Open `business-insights.xlsx` in Excel, Apple Numbers, or Google Sheets.",
    "  Start with the **README** tab — it explains what every column means.",
    "",
    "## Need the raw data?",
    'A separate "Download raw JSON (for engineers)" button in the app',
    "produces a `portfolio.json` file with the full structured dataset.",
    "",
  ].join("\n");
}

/**
 * Engineer-side JSON download body. Byte-equivalent to the prior bundle's
 * portfolio.json — `JSON.stringify(portfolio, null, 2)`. Used by the
 * secondary "Download raw JSON (for engineers)" button (plan 03-04).
 *
 * Public so tests can assert byte equivalence with the prior bundle.
 */
export function buildPortfolioJson(portfolio: PortfolioStats): string {
  return JSON.stringify(portfolio, null, 2);
}

/**
 * Engineer-side JSON download filename. Mirrors `packFilename`'s slug rule
 * so the on-disk artifact stays recognizable next to its zip cousin.
 *
 * `"Real Data"` + 2026-04-22 → `portfolio-real-data-2026-04-22.json`.
 * Empty / whitespace label → `portfolio-pack-2026-04-22.json` (same fallback
 * `packFilename` uses).
 *
 * Public so `BusinessInsightsView.test.tsx` (plan 03-04) can assert the
 * filename pattern when the engineer-side button fires.
 */
export function jsonFilename(
  datasetLabel: string,
  generatedAt: Date = new Date(),
): string {
  const slug = datasetLabel
    .toLowerCase()
    .replace(/[^a-z0-9]+/g, "-")
    .replace(/^-+|-+$/g, "");
  const date = generatedAt.toISOString().slice(0, 10);
  return `portfolio-${slug || "pack"}-${date}.json`;
}
```
