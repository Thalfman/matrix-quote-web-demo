## Why this plan

CONTEXT D-07 locks the engineer-side affordance: a secondary button labeled exactly **`Download raw JSON (for engineers)`** placed next to the primary "Download insights pack" button. Triggers a download of `portfolio.json` only (single file, NOT zipped). Visually de-emphasized (`text-xs` / `text-muted` / underline-on-hover) so a non-technical reviewer reads the parenthetical disclaimer and ignores it. Filename: `portfolio-{slug}-{date}.json`.

This satisfies the "moved to a separate 'for engineers' download path" alternative in ROADMAP success #1 — engineers who consumed `portfolio.json` from the prior bundle keep working with one extra click; non-technical reviewers see the warning and never accidentally open a JSON file.

## Risks / pitfalls

- **R1 — Disabled-state correctness.** D-07 says "shares the same disabled state semantics (no race with the primary build)". Both buttons use the same `isPackBuilding` state. While `isPackBuilding === true`, BOTH are disabled. Don't introduce a separate `isJsonBuilding` flag — JSON building is synchronous (string-only), but disabling it during pack build prevents any race where the user double-fires. T2 asserts: while pack is building, JSON button is disabled too.
- **R2 — JSON build is synchronous; no need for `useState` to track it.** `buildPortfolioJson` is pure `JSON.stringify`. The download blob creation is also synchronous. The handler is a sync `useCallback`, NOT `async`. T2 asserts the click triggers `downloadBlob` synchronously.
- **R3 — `downloadBlob` is shared.** Use the existing `downloadBlob` export — do not re-implement. Make the JSON Blob with `new Blob([json], { type: "application/json;charset=utf-8;" })`.
- **R4 — Filename slugification.** Reuse the existing `packFilename` slug logic. Since `packFilename` returns `business-insights-{slug}-{date}.zip`, we need a JSON-flavored sibling. Two options: (a) export a new `jsonFilename(datasetLabel, generatedAt)` helper from `exportPack.ts`; (b) inline the same slug logic at the call site. **Choose (a)** — it keeps the slug logic single-source. Add `jsonFilename` to `exportPack.ts` in T1's first sub-step (this plan does NOT touch the workbook/zip helpers, so this single-line addition to exportPack.ts is in scope here).
- **R5 — Layout.** The current container is `<div className="ml-auto pb-2">` holding only the primary button. Wrap both buttons in `<div className="ml-auto pb-2 flex items-center gap-3">` (or similar). On narrow screens, the parent flex-wrap row already handles overflow. Don't add a new responsive breakpoint — keep visual diff small.
- **R6 — Styling: text-xs / text-muted / underline-on-hover.** D-07 explicitly. Use Tailwind: `text-xs text-muted hover:text-ink hover:underline focus-visible:outline-none focus-visible:ring-2 focus-visible:ring-teal disabled:opacity-50 disabled:cursor-not-allowed`. No icon (the primary button has the `Download` icon; secondary stays text-only to look subordinate).
- **R7 — A11y posture matches Phase 2.** Native `<button type="button">` (focusable by default), `disabled={isPackBuilding}`, `aria-label="Download raw portfolio JSON for engineers"` (gives screen-reader users the full disclosure even though the visible label is the same text). Phase 2 used `focus-visible:ring-teal` on glossary affordances; same token here.
- **R8 — Zero ML jargon in new copy.** "Download raw JSON (for engineers)" is the only new on-screen string. "raw JSON" and "for engineers" are not on the banned list. The hover-to-show `title` attribute on the button is a non-jargon explanation: `title="Download the full structured portfolio data as a single JSON file."`. T2's jargon-guard `describe` asserts the rendered DOM is jargon-free.
- **R9 — Existing tests preserved.** The tab/filter/drawer/loading tests in `BusinessInsightsView.test.tsx` (lines 54-336) must continue to pass unchanged. T2 only ADDS a new `describe("BusinessInsightsView - download buttons (INSIGHTS-01)", …)` block at the end; no edits to the existing blocks.
- **R10 — `vi.spyOn` approach for the download trigger.** The existing tests don't currently mock `downloadBlob`. The new tests need to assert that clicking the JSON button calls `downloadBlob` with a JSON-typed Blob and the correct filename. Use `vi.spyOn(exportPackModule, "downloadBlob")` — but `downloadBlob` is a named export, so the canonical pattern is `vi.mock("./exportPack", async () => { const actual = await vi.importActual<…>("./exportPack"); return { ...actual, downloadBlob: vi.fn() }; })`. This preserves all real exports and replaces only `downloadBlob` with a spy.
